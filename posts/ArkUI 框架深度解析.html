1. 框架概览与分层架构

ArkUI 是 OpenHarmony 生态中核心的 UI 渲染框架，采用声明式开发范式。开发者通过 ArkTS 语言描述界面，框架负责组件树构建、布局测量、渲染绘制及事件处理。整个框架可划分为三个核心层次：

┌─────────────────────────────────────────────────────────┐
│                    声明层 (ArkTS API)                    │
│  @State / @Prop / @Link 等装饰器，开发者编写声明式 UI    │
├─────────────────────────────────────────────────────────┤
│                    中间层 (组件树与差分更新)              │
│  Component 树 → Element 树 → FrameNode 树 → RSNode 树   │
│  状态管理、脏标记系统、差异计算、布局测量调度             │
├─────────────────────────────────────────────────────────┤
│                    渲染层 (Native 渲染管线)              │
│  RenderService 进程：指令录制、GPU 合成、屏幕显示        │
└─────────────────────────────────────────────────────────┘
  • 声明层：开发者使用 ArkTS 语言配合状态装饰器（@State、@Prop、@Link 等）描述 UI 结构，build() 方法定义了组件树的静态描述。
  • 中间层：框架将声明式的 UI 描述转换为内部的组件树，并在状态变更时执行差分更新算法，标记脏节点，决定哪些部分需要重新测量、布局或绘制。核心代码位于 foundation/arkui 的 C++ AceEngine 中。
  • 渲染层：将更新后的节点树转换为平台原生绘制指令，通过 IPC 提交给 RenderService 进程进行 GPU 合成并最终上屏。

2. 核心节点模型：三棵树

ArkUI 底层维护了三棵关键的树形结构，以实现高效的声明式 UI 更新：

树结构 作用 说明
Component 树 前端 UI 组件的映射 每个 UI 组件对应一个 Component 对象，描述组件的配置信息
Element 树 维护组件的树形结构和更新状态 由 Component 生成，管理组件生命周期和状态依赖关系
FrameNode 树 UI 描述的数据结构，生成渲染树 负责测量、布局、属性计算，是 UI 树的“骨架”
RSNode 树 录制渲染命令 将绘制指令 IPC 发送至 RenderService 进程完成绘制

编译期间，装饰器修饰的 UI 组件树被解构成 ViewPU 树；运行时响应 Vsync 信号，进一步构建 FrameNode 树和 RSNode 树。

3. 渲染流水线：Build → Measure → Layout → Render

一次完整的 UI 更新需经历以下四个阶段：

状态变更 (State Change)
      ↓
【Build】执行组件创建和组件标脏
      ↓
【Measure】对组件的宽高进行测量
      ↓
【Layout】在屏幕上确定元素的位置
      ↓
【Render】提交绘制指令
      ↓
屏幕显示

3.1 Build 阶段

Build 是执行组件创建和组件标脏的过程。当状态变量（如 @State 修饰的数据）发生变化时，框架会调用所在组件的 build() 方法。

关键点：

  • build() 方法并不直接绘制 UI，而是根据当前状态描述 UI 应该是什么样子。
  • 框架会将新旧 UI 描述进行对比，仅对发生变化的属性进行标脏，为后续的测量布局做准备。

3.2 Measure 阶段

Measure 是对组件宽高进行测量的阶段：

  1. 父节点传递约束：父组件将 LayoutParam（约束条件）传递给子组件。
  2. 子节点自底向上计算：子组件根据约束计算自身所需尺寸。
  3. 递归遍历 children：框架根据脏节点数组递归遍历子节点进行测量。若布局参数未变化，则跳过该阶段。

在源码层面，Measure 通过 RenderNode::PerformLayout() 中的 Measure() 方法实现，最终调用 SetLayoutSize() 确定尺寸。

3.3 Layout 阶段

Layout 是对元素在屏幕上进行位置摆放的阶段。当 Measure 执行完成后，框架会根据布局算法计算出每个 RenderNode 在相对空间中的真实位置。

布局触发机制：节点内容变化时会标记自己为 needLayout，并向上标记至重绘边界（RepaintBoundary）。从重绘边界开始，父节点将 LayoutParam 传给子节点，子节点据此计算尺寸和位置。

3.4 Render 阶段

Render 是根据测量和布局得到的大小、位置信息，提交绘制指令的过程：

  1. RSNode 录制绘制命令
  2. 绘制命令通过 IPC 发送至 RenderService 进程
  3. Graphics 系统完成合成并上屏

ArkUI 区分两类“脏”状态：

脏类型 影响范围 处理方式
布局脏 影响尺寸和位置 需重新测量布局，并判定影响范围
绘制脏 仅影响样式（颜色、背景等） 重绘但不重新布局

4. 状态管理与 Build 触发条件

4.1 Build 方法何时被调用？

核心规则：当组件或其子组件直接依赖的状态变量发生变化时，才会触发该组件的 build() 方法重新执行。

具体触发条件包括：

  • @State 变量变化：调用所在组件的 build()。
  • @Prop / @Link 变量变化：父组件关联数据变化时，子组件同步更新并触发 build()。
  • @Provide / @Consume 数据变化：跨层级状态共享时触发相关组件刷新。
  • @ObjectLink / @Observed：当 @Observed 修饰的类对象属性变化时，触发依赖该属性的组件 build()。

4.2 数据驱动机制

框架在运行时维护组件树，通过装饰器实现状态观察。状态变更触发依赖收集，精准标记相关组件为脏，在布局过程中只更新需要刷新的组件，避免造成整棵组件树的重建。

4.3 更新粒度与差分机制

ArkUI 的更新粒度由“脏节点数组”控制。Build 过程按照组件 ID 依次更新组件设置的属性，框架通过计算新旧 UI 描述之间的差异，只对发生变化的节点执行后续的 Measure/Layout/Render 流程。

5. 触发子组件 Build 的详细场景示例

✅ 会触发子组件 build() 的情况

场景 会触发的组件 代码示例
1. 子组件自身的 @State 变量改变 该子组件自身 @Component struct Child { @State count = 0; build() { Text(${this.count}).onClick(()=> this.count++) } }
2. 父组件的 @State 改变，子组件 @Prop 依赖 该子组件 Child({ propCount: this.parentCount })
3. 任何一端改变 @Link 绑定的变量 所有绑定的父子组件 @Link linkCount: number; 且父子组件通过 $ 语法建立双向绑定
4. 跨层级 @Provide 变量改变，后代组件 @Consume 依赖 所有依赖该状态的@Consume组件 祖先组件：@Provide("key") count = 0;
后代组件：@Consume("key") count: number;
5. @Observed 类实例的属性改变，且被 @ObjectLink 引用 引用了该实例的子组件 @ObjectLink user: User; 且 User 类被 @Observed 装饰

❌ 不会触发子组件 build() 的情况（常见误区）

场景 说明 代码示例
1. 修改非状态变量 只有被@State等装饰器修饰的变量才是响应式的。修改普通变量UI不会刷新。 private count = 0; build() { Text(${this.count}).onClick(()=> this.count++) // 无效！ }
2. 修改 @Prop 但父组件状态未变 @Prop是单向传递。子组件内修改它，自身UI会变，但不会影响父组件，且若父组件因其他原因重建，子组件的修改会被覆盖。 Child({ title: this.parentTitle }) 子组件内 this.title = "New" 只在本地生效。
3. 修改 @ObjectLink 变量自身（而非其属性） @ObjectLink 是只读引用，不能直接对变量本身赋值，否则运行时报错。 this.objLink = new User() ❌
4. 修改非 @Observed 类实例的属性 对于嵌套对象，若类未被@Observed装饰，框架无法观测其属性变化。 假设User类无@Observed，修改 user.name = "新名字" 不会触发UI更新。
5. 直接修改数组的索引或length属性 部分框架版本对此观测有限，推荐使用push、pop等能产生新引用的数组方法。 this.arr[0] = 5 可能不更新，推荐 this.arr = [...this.arr]。
6. 数据变更不触发 build 的刷新边界 如果状态变量仅在未被使用的组件中更新，或框架能判断出UI无需重绘，则相关组件的 build 也不会执行。 在一个独立组件中更新其内部状态，但该状态不影响任何UI元素的属性或内容。

6. 父组件尺寸变化后子组件的更新机制

6.1 更新流程

当父组件尺寸发生变化时，子组件的更新遵循以下流程：

  1. 属性变化检测：父组件的布局属性（width、height、padding、margin 等）改变时，框架将其标记为“布局脏”。
  2. 布局边界查找：框架向上查找到布局边界（设置了固定宽高尺寸的组件），边界内部的布局变化不会影响外部。
  3. 子树更新：从布局边界开始，对受影响的子树重新测量和布局：
    • 父节点将新的 LayoutParam（约束条件）传递给子节点。
    • 子节点根据新约束重新计算自身尺寸和位置。
  4. 渲染更新：根据测量和布局得到的新大小位置信息，提交绘制指令完成渲染。

6.2 响应式布局示例

以 Grid 组件为例，当网格容器组件尺寸发生变化时，所有子组件以及间距会等比例调整，实现网格布局的自适应能力。

6.3 监听尺寸变化

开发者可以通过 onAreaChange 事件监听组件尺寸变化，该事件会在组件由布局变化导致的尺寸/位置变化时触发。注意：由绘制变化（如 translate、offset）导致的渲染属性变化不会触发该回调。

8. 性能优化要点

8.1 利用布局边界减少重算

对固定尺寸的组件设置具体宽高，使其成为布局边界，限制布局变化的影响范围，避免整棵树重新布局。

8.2 精简组件数量

使用扁平化布局组件（如 RelativeContainer、Grid）替代多层 Column/Row 嵌套，减少中间节点数量。

8.3 使用 LazyForEach 懒加载

在滚动列表中，使用 LazyForEach 仅渲染可视区域内的项，减少一次性组件创建数量。

8.4 区分布局属性与样式属性

  • 布局属性（width、height、padding、margin）变化会触发子树重新测量布局，成本较高。
  • 样式属性（color、backgroundColor、opacity）仅影响自身，不会进行子树查找，成本较低。

8.5 使用无状态 Builder 替代 @Component

优先使用无状态组件 @Builder 替代 @Component，减少状态依赖和更新开销。

8.6 数组更新最佳实践

  • 尽量使用返回新数组的方法（如 map、filter、slice、扩展运算符 ...）。
  • 避免直接修改数组索引或 length，以确保状态观测的可靠性。
  • 对于 @Observed 类的数组属性，需结合 @ObjectLink 使用。

9. 总结

ArkUI 通过声明式开发范式、三棵树模型和精细的差分更新算法，为开发者提供了高效、稳定的 UI 开发体验。理解 Build → Measure → Layout → Render 的渲染流水线，以及状态装饰器如何触发组件更新，是编写高性能 ArkUI 应用的基础。通过合理设计组件边界、区分布局与样式属性、善用懒加载等优化手段，可以进一步提升应用的流畅度和响应速度。
